# Anonymous Payroll Reporter — Tools & Fixtures

A monorepo for the Anonymous Payroll Reporter PWA and its test/fixture tooling. The PWA runs fully client-side, extracting payroll and pension contribution data and rendering a printable report. This repo also includes the Python/Node pipelines that generate synthetic PDF and Excel fixtures used by the test suite.

Both the fixture and PWA parsing tooling are designed to support multiple payroll providers and pension schemes through configuration and format files, but have currently only been tested against payroll PDFs generated by [Sage Payroll](https://www.sage.com/en-gb/sage-business-cloud/payroll/) and contribution reports generated by the [NEST pension scheme](https://www.nest.org/).

## Repository layout

- **`pwa/`** — Client-side PWA (Vue + PDF.js + service worker)
- **`generate_fixtures/`** — PDF/Excel fixture generators (Python)
- **`tests/`** — Vitest suite covering PDF parse, contribution parse, and report workflows

### PWA asset generation (icons + splash screens)

Generate PWA icons and iOS splash screens from the source SVG using the `pwa:generate` script:

```bash
pnpm pwa:generate
```

> **Why the script calls `node` directly on the CLI path rather than `pnpm exec pwa-asset-generator`:**
> `pwa-asset-generator@8.1.2` has a known bug ([#1268](https://github.com/elegantapp/pwa-asset-generator/issues/1268)) where `puppeteer-core@24.x` crashes with `Target.createTarget: Target closed` when generating images concurrently. A patch is applied via `patches/pwa-asset-generator.patch` (registered in `pnpm.patchedDependencies`) that switches screenshot capture from a parallel `Promise.all` to a sequential loop reusing a single browser context and page.
>
> However, pnpm skips re-applying patches when the package is already extracted in its content-addressed store (i.e. `pnpm install` reports "Already up to date"). Calling `pnpm exec pwa-asset-generator` would run the unpatched binary and crash. Calling `node` directly on the patched file in `node_modules/.pnpm/` bypasses this.
>
> If you run `pnpm install` and `pnpm pwa:generate` crashes again, re-apply the patch manually:
>
> ```bash
> cp node_modules/.pnpm_patches/pwa-asset-generator@8.1.2/dist/helpers/puppets.js \
>    node_modules/.pnpm/pwa-asset-generator@8.1.2/node_modules/pwa-asset-generator/dist/helpers/puppets.js
> ```

## Pre-commit & CI workflows

### Pre-commit (local)

Husky runs the pre-commit workflow automatically before every commit.
You can also run it manually:

```bash
pnpm precommit
```

This runs, in order:

1. `pnpm format` (Prettier)
2. `pnpm jslint` (ESLint)
3. `pnpm csslint` (Stylelint)
4. `pnpm check:js` (TypeScript)
5. `pnpm fixtures:check` (fixture drift check)
6. `pnpm fixtures:expected-pdf` (refresh PDF parse snapshots)
7. `pnpm fixtures:expected-excel` (refresh Excel parse snapshots)
8. `pnpm pwa:build` (Vite production build)
9. `pnpm test:all` (Vitest)

### CI (GitHub Actions)

CI mirrors the same checks as `pnpm precommit` to keep local and CI expectations aligned.

## Releases (release-please)

This repo uses [release-please](https://github.com/googleapis/release-please) via GitHub Actions (`.github/workflows/release-please.yml`).

### How it works

- On every push to `main`, release-please opens or updates a **release PR**.
- When a release PR is merged, release-please creates a GitHub release and tag.

### Automated deploy on release

If a release is created, the workflow automatically:

1. Checks out the repo
2. Installs Node dependencies
3. Runs `pnpm test:all`
4. Deploys to Cloudflare Workers via `wrangler-action@v3`

No manual deploy is required after a release PR merge, assuming the workflow succeeds.

## Deployment

The PWA is deployed to Cloudflare Workers using [Wrangler](https://developers.cloudflare.com/workers/wrangler/). A minimal Worker script (`src/worker.js`) sets `Cache-Control: no-cache` on `sw.js`, `index.html`, and `site.webmanifest` so browsers always check for PWA/service worker updates; all other static assets in `pwa/` are served directly from Cloudflare's edge.

### First-time setup

Install Wrangler globally (one-time):

```bash
npm install -g wrangler
```

Authenticate with Cloudflare (one-time):

```bash
wrangler login
```

### Deploy to production

```bash
pnpm cf:deploy
```

Wrangler bundles `src/worker.js`, uploads the `pwa/` directory as static assets, and prints the live URL on success.

To force a fresh production build before deploy, use:

```bash
pnpm cf:deploy:build
```

### Upload a preview version (no live traffic)

```bash
pnpm cf:preview
```

To build first, use:

```bash
pnpm cf:preview:build
```

### Custom domain

The app is served at `https://payroll-beta.fairtools.work`. The `routes` entry in `wrangler.jsonc` binds this to the `fairtools.work` zone in Cloudflare — the domain must be active in your Cloudflare account before deploying.

## Releases

This repo uses [release-please](https://github.com/googleapis/release-please) for automated versioning. On every push to `main`, the release-please GitHub Action maintains an open release PR. Merging it:

- bumps the version in `package.json`
- bumps `CACHE_NAME` in `pwa/sw.js` (invalidating the Service Worker cache for all users)
- creates a tagged GitHub release with a generated changelog

Version increments are driven by [conventional commits](https://www.conventionalcommits.org/): `fix:` → patch, `feat:` → minor, `feat!:` / `BREAKING CHANGE:` → major.

See `pwa/README.md` for the full deployment checklist.

## Update reminders (important)

- **Service Worker cache invalidation:**
  `CACHE_NAME` in `pwa/sw.js` is bumped automatically when the release-please PR is merged. No manual change needed.

- **Regenerate fixtures when inputs change:**
  If `generate_fixtures/payroll_inputs.json` or structure configs change, regenerate fixtures and expected snapshots before running tests.
